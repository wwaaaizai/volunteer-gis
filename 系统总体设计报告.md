## 一、引言

### 1.1 系统背景

随着高校学生活动的日益丰富，传统的纸质报名和人工管理方式已难以满足大规模志愿活动的组织需求。中国矿业大学每年举办大量的志愿活动，涵盖迎新服务、图书馆整理、校运会协助、社区服务等多种类型。组织者需要高效地发布活动信息、管理报名数据、跟踪签到情况，而学生则希望能够方便地浏览活动、完成报名和签到。

WebGIS技术的快速发展为校园活动管理提供了新的思路。通过将活动信息与地理空间位置相结合，学生可以直观地在地图上查看活动分布，组织者可以通过地图选点精确设定活动地点。基于此背景，本项目开发了"基于GIS的校园志愿活动服务系统"，旨在为中国矿业大学师生提供一个集活动发布、地图浏览、报名管理、GPS签到签退于一体的WebGIS平台。

### 1.2 系统目标

本系统的总体目标是为中国矿业大学构建一个完整、易用的校园志愿活动管理平台。具体目标包括：

（1）实现志愿活动的基本管理功能，包括活动的创建、编辑、发布、搜索，支持活动状态的自动流转（草稿→已发布→进行中→已结束）；

（2）基于天地图WMTS瓦片服务，实现活动信息的地图可视化展示，支持地图选点、多点选取和POI地址自动识别；

（3）实现完整的报名管理流程，学生可以通过地图或列表浏览活动并报名，组织者可以审核报名、查看签到统计；

（4）支持GPS定位签到和签退，通过Haversine距离计算验证签到有效性，自动计算志愿时长；

（5）引入大语言模型辅助活动文案生成，提升组织者的工作效率；

（6）提供GIS分析功能，包括活动热力图、缓冲区分析、覆盖率统计、集合点推荐等。

### 1.3 定义

本报告涉及的专有名词和缩写定义如下：

- GIS：地理信息系统（Geographic Information System）
- WebGIS：基于Web的地理信息系统
- GCJ-02：国测局坐标系（火星坐标系），中国境内地图服务使用的加密坐标系
- WGS-84：世界大地测量系统1984，GPS使用的标准坐标系
- WMTS：Web地图瓦片服务（Web Map Tile Service）
- WFS：Web要素服务（Web Feature Service）
- JWT：JSON Web Token，用于无状态身份认证
- POI：兴趣点（Point of Interest），在地图上标注的位置点
- Haversine：一种计算球面两点距离的公式

## 二、需求分析

### 2.1 需求规定

本系统的功能需求按子系统划分如下：

**（一）用户与权限子系统。** 支持学生注册登录、组织者申请与审批、管理员账号管理。采用JWT+BCrypt实现无状态认证，使用Spring Security @PreAuthorize注解进行方法级权限控制。角色分为student（学生）、organizer（组织者）、admin（管理员）三种。

**（二）活动创建与审核子系统。** 支持活动的完整CRUD操作。创建活动时支持通过MapPicker组件在地图上选取活动地点，支持多点选取（一个活动关联多个地理位置），支持POI自动识别建筑物名称，支持AI辅助生成活动标题和描述，支持活动模板（预设年常活动模板+自定义模板）。编辑活动时按状态分级控制权限（草稿可全字段编辑，已发布仅可修改描述）。

**（三）活动地图子系统。** 支持在天地图底图上以GeoJSON格式展示活动标注点，点击标注可查看活动详情并报名。支持GeoServer WFS矢量图层叠加（校园建筑、运动场等），支持建筑点击查询。支持活动热力图、缓冲区分析等GIS分析功能。

**（四）报名管理子系统。** 学生可报名/取消报名，系统进行数据库级名额校验（防止超额报名）。组织者可查看报名名单，支持审核报名（通过/拒绝）。提供报名统计指标（总计/待审核/已通过/已拒绝/已签到/已签退）。

**（五）签到签退子系统。** 支持GPS定位签到（通过Haversine公式计算与活动地点的距离，500米内允许签到），支持二维码签到，支持签退并自动计算志愿时长。支持签到地理围栏（不规则多边形签到区域）。

**（六）站内信子系统。** 系统自动发送报名确认、审核结果等通知消息。

### 2.2 运行环境

本系统采用浏览器/服务器（B/S）架构，用户通过浏览器访问系统，服务器端处理业务逻辑和数据存储。

**（一）硬件环境：**

服务器端：普通PC或云服务器，CPU 2核以上，内存8GB以上，硬盘100GB以上。

客户端：支持现代浏览器的PC或移动设备，推荐Chrome 90+或Edge 90+。

**（二）软件环境：**

服务器操作系统：Windows 10/11 或 Linux（支持JDK 17的运行环境）。

数据库：MySQL 8.0 Community Server，InnoDB存储引擎，utf8mb4字符集。

应用服务器：Spring Boot 3.2.0内嵌Tomcat，无需额外部署Web服务器。

GIS服务：GeoServer 2.24+，天地图WMTS瓦片服务。

开发工具：IntelliJ IDEA（后端）、VS Code（前端）、MySQL Workbench（数据库管理）、Git（版本控制）。

**（三）网络环境：**

系统运行于校园局域网或互联网环境。天地图瓦片服务和GeoServer服务需要网络可达。开发阶段支持使用MSW（Mock Service Worker）进行纯前端离线开发。

### 2.3 设计概述和处理流程

系统采用前后端分离架构。前端为Vue 3单页应用（SPA），通过Axios发送HTTP请求与后端交互。后端为Spring Boot RESTful API服务，通过MyBatis-Plus操作MySQL数据库。

核心业务流程如下：

**学生视角：** 打开系统→浏览活动地图→点击活动查看详情→在线报名→活动当天到现场→GPS定位签到→活动结束签退→系统自动计算志愿时长。

**组织者视角：** 登录组织者账号→进入仪表盘→创建活动（地图选点/模板/AI辅助）→发布活动→查看报名名单→审核报名→活动结束后查看签到统计。

**管理员视角：** 登录管理员账号→管理全部活动→审批组织者申请→管理报名数据→审核时长。

**技术处理流程：** 前端通过MapLibre GL JS渲染天地图底图→用户点击地图→获取GCJ-02坐标→转换为WGS-84→存入数据库→需要展示时再从WGS-84转回GCJ-02→渲染到地图上。

### 2.4 系统结构

系统采用"五子系统单体架构+geo横切模块"的设计模式。各子系统在同一个Spring Boot应用中按包划分，通过Spring的依赖注入实现模块间的松耦合。前端对应使用Vue Router按路由划分页面，Pinia管理全局状态。

系统分层架构从上到下为：

表现层：Vue 3 + Element Plus组件库 + MapLibre GL JS地图引擎。负责用户界面展示和人机交互。

控制层：Spring Boot的@RestController，负责接收HTTP请求、参数校验、调用服务层、返回统一响应。

服务层：各子系统的Service接口和实现类，负责核心业务逻辑。

数据访问层：MyBatis-Plus的BaseMapper，负责数据库的CRUD操作。

数据存储层：MySQL 8.0数据库，存储用户、活动、报名、签到等业务数据。

GIS服务层：SpatialCalculator（空间计算）、GeoJsonBuilder（GeoJSON构造），为地图和签到模块提供GIS能力支撑。

## 三、接口设计

### 3.1 用户接口

系统提供Web浏览器作为用户操作界面。主要页面包括：登录页面（表单输入学号和密码）、注册页面（填写个人信息）、地图主页面（天地图底图+活动标注+底部活动预览背板）、活动详情页面（展示活动信息+报名按钮）、活动创建页面（表单+地图选点+AI生成入口）、组织者仪表盘页面（四状态Tab+活动列表+操作按钮）、管理员后台页面（活动管理+组织者审批）、签到页面（GPS定位+按钮操作）、个人中心页面（我的报名+志愿足迹）。

页面布局采用响应式设计，桌面端使用侧边导航栏+主内容区布局，移动端自动切换为底部Tab导航栏。界面风格统一使用Element Plus组件库浅色主题。

主要前端路由如下：

```
/                  → 活动地图主页
/login             → 登录页
/register          → 注册页
/admin             → 管理员后台
/organizer         → 组织者仪表盘
/organizer/create  → 创建活动
/activity/:id      → 活动详情
/my-signups        → 我的报名
/checkin           → 签到签退
/my-footprint      → 志愿足迹
```

### 3.2 外部接口

**（1）天地图WMTS接口。** 前端通过MapLibre GL JS的raster source类型加载天地图WMTS瓦片。请求格式为HTTPS GET，URL模板为`https://t0.tianditu.gov.cn/vec_w/wmts?SERVICE=WMTS&REQUEST=GetTile&...`，参数包括tk（API Key）、TILEMATRIX（缩放级别z）、TILEROW（行y）、TILECOL（列x）。天地图使用GCJ-02坐标系。

**（2）GeoServer WFS接口。** 前端通过WFS GetFeature请求获取校园矢量图层数据。请求通过Vite代理转发至GeoServer（端口8338），workspace为CUMT。返回数据为GeoJSON格式，前端通过wgs84ToGcj02函数进行坐标变换后叠加到天地图底图上。

**（3）智谱AI接口。** 后端通过HTTPS POST调用智谱开放平台API（open.bigmodel.cn），使用glm-4-flash模型进行文本生成。请求头携带Bearer Token认证，请求体包含model、temperature（0.9）和messages数组（system+user角色）。

**（4）高德地图API接口。** 用于GIS分析中的路径规划功能。通过后端代理调用高德地图Web服务API，解决浏览器端CORS跨域限制。

### 3.3 内部接口

系统内部接口主要指前后端之间的RESTful API通信。前端通过Axios实例发送请求，Vite开发服务器将`/api/*`路径代理转发至后端Spring Boot服务（端口9090），`/uploads/*`路径代理至后端静态资源目录。

后端主要REST接口汇总：

**认证模块（/api/auth）：** POST /register（注册）、POST /login（登录返回JWT）、GET /me（获取当前用户信息）。

**活动模块（/api/activities）：** GET /（公开活动列表）、GET /{id}（详情）、POST /（创建，admin/organizer权限）、PUT /{id}/publish（发布）、PUT /{id}（编辑）、GET /my（组织者视角）、GET /search（关键词搜索）。

**地图模块（/api/map）：** GET /activities（返回活动GeoJSON FeatureCollection）、GET /layers（图层清单）。

**报名模块（/api/signups）：** POST /（报名）、DELETE /（取消）、GET /my（我的报名）、GET /activity/{id}（活动报名名单）、PUT /{id}/review（审核）。

**签到模块（/api/checkin）：** POST /location（GPS签到）、POST /qr（二维码签到）、POST /out（签退）。

**AI模块（/api/ai）：** POST /generate-description（生成标题和描述）、POST /generate-cover（生成封面）。

**模板模块（/api/templates）：** GET /（列表）、POST /（保存）、DELETE /{id}（删除）。

**上传模块（/api/upload）：** POST /image（上传封面图）。

后端统一响应格式：`{"code":200,"message":"success","data":{...}}`。成功时code为200，业务异常时code为500，权限不足时返回401或403。

## 四、模块设计

**（四）报名管理模块（ABM）**

后端：SignupController（报名/取消/审核/查询）、SignupService/SignupServiceImpl（事务控制，名额校验，审核逻辑）。

前端：ActivityDetail.vue（活动详情+报名按钮）、MySignups.vue（我的报名列表）。

**（五）签到签退模块（APM）**

后端：CheckInController（GPS签到/二维码签到/签退）、CheckInService（Haversine距离校验，时长计算=签退时间-签到时间）。

**（六）GIS公共模块（GEO）**

后端：SpatialCalculator（Haversine距离计算、isWithinCampus校区校验、isPointInPolygon围栏判断）、GeoJsonBuilder（GeoJSON构造）。

### 4.2 运行控制

**（1）前端启动流程：** 浏览器访问localhost:5173→Vite开发服务器返回index.html→加载main.ts→根据VITE_USE_MOCK配置决定是否启动MSW Service Worker→创建Vue应用实例→挂载Pinia和Router→渲染App.vue→路由守卫检查登录状态→重定向到登录页或首页。

**（2）后端启动流程：** 执行mvnw.cmd spring-boot:run→Spring Boot自动配置→扫描@Component注册Bean→DataInitializer（CommandLineRunner）启动时检查数据库，如无数据则自动创建测试账号和演示活动→嵌入Tomcat监听9090端口→等待HTTP请求。

**（3）请求处理流程：** 前端Axios发起请求→Vite代理转发至9090→Spring Boot DispatcherServlet路由到对应Controller→JwtAuthFilter验证Token→@PreAuthorize检查权限→Service层处理业务→MyBatis-Plus操作MySQL→Controller返回Result统一响应体→Axios拦截器提取data→Vue组件更新视图。

**（4）地图渲染流程：** 用户打开地图页面→useMap composable创建MapLibre实例→加载buildTiandituStyle()构造的样式（含天地图WMTS raster source）→map.on('load')后加载ActivityLayer→ActivityLayer请求/api/map/activities→后端返回GeoJSON FeatureCollection→前端wgs84ToGcj02转换→addSource/addLayer渲染标注点→监听地图交互事件（点击标注→跳转活动详情，点击建筑→WFS查询弹出信息）。

### 4.3 关键算法

**（1）Haversine距离计算。** 用于签到距离校验和POI匹配。公式利用地球球面三角关系计算两点间的最短弧长距离，输入为两点经纬度，输出为米。在签到场景中，阈值设为500米；在POI匹配场景中，阈值设为150米。

```javascript
function haversineMeters(lng1, lat1, lng2, lat2) {
  const R = 6371000  // 地球半径（米）
  const dLat = (lat2 - lat1) * Math.PI / 180
  const dLng = (lng2 - lng1) * Math.PI / 180
  const a = Math.sin(dLat/2) ** 2
    + Math.cos(lat1*PI/180) * Math.cos(lat2*PI/180) * Math.sin(dLng/2) ** 2
  return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1-a))
}
```

**（2）坐标转换（WGS-84 ↔ GCJ-02）。** 天地图使用GCJ-02坐标系，GPS和数据库使用WGS-84，两者在中国境内有约300-500米偏移。正向转换（WGS→GCJ）使用国测局公开算法。反向转换（GCJ→WGS）使用迭代逼近法，5轮收敛。

**（3）JWT令牌生成与验证。** 使用HMAC-SHA512算法（HS512）对用户信息（userId、studentId、role）签名，设置24小时过期时间。每次请求从Authorization头提取Bearer Token，解析后设置SecurityContext。

**（4）K-Means++集合点推荐。** 后端MapService实现，从报名学生地理位置使用K-Means++算法聚类，输出最优集合点坐标。

## 五、系统数据结构设计

### 5.1 逻辑结构设计

系统核心数据实体及关系如下：

**用户（User）。** 存储学生、组织者和管理员的基本信息。主键id，唯一键student_id（学号）。关键字段：password（BCrypt加密）、name、phone、grade（年级）、college（院系）、role（student/organizer/admin）、organization（组织者所属机构）、employee_id（组织者工号）、total_hours（累计志愿时长）。

**活动（Activity）。** 存储志愿活动的完整信息。主键id，外键creator_id和organizer_id关联User表。关键字段：title、description、location_name、longitude/latitude（WGS-84）、start_time/end_time、max_participants/signed_count、status（draft/published/ongoing/ended/cancelled）、category、tags、cover_image、extra_locations（JSON，分地点列表）、checkin_region（TEXT，签到围栏GeoJSON）。

**报名记录（Signup）。** 存储学生报名信息。主键id，外键activity_id和user_id。关键字段：status（signed/approved/rejected/signed_in/signed_out/cancelled）、sign_in_time/sign_in_lng/sign_in_lat、sign_out_time、volunteer_hours、review_reason。

**站内信（Message）。** 系统通知消息。主键id，外键user_id。关键字段：title、content、type（system/signup/signin）、is_read。

**组织者申请表（OrganizerApply）。** 组织者注册申请记录。主键id，外键user_id和reviewed_by。关键字段：organization、reason、status（pending/approved/rejected）。

**操作日志（OperationLog）。** 记录管理员和组织者的关键操作。主键id，外键user_id和activity_id。关键字段：operation_type、description。

### 5.3 数据结构与程序的关系

前后端通过RESTful API传输JSON格式数据。Java实体类通过MyBatis-Plus的@TableName注解与数据库表映射，字段命名采用驼峰命名法（如locationName），数据库列名采用下划线命名法（如location_name），MyBatis-Plus自动完成转换（map-underscore-to-camel-case: true）。

前端TypeScript的interface定义（如MockActivity、MockSignup）与后端实体结构保持一致，通过Axios拦截器自动提取data字段，前端组件直接使用响应数据进行渲染。分地点数据（extra_locations）以JSON数组格式存储，前端通过JSON.stringify序列化后提交，编辑时通过JSON.parse反序列化回填。

关键的数据流：用户在前端地图点击→MapLibre返回GCJ-02坐标→gcj02ToWgs84转为WGS-84→存入数据库longitude/latitude字段→需要展示时从数据库读取→wgs84ToGcj02转为GCJ-02→渲染到天地图底图上。

## 六、系统出错处理设计

### 6.1 出错信息

系统可能出现的错误类型及处理方式如下：

（1）**网络错误：** 前端Axios拦截器捕获网络异常，显示"网络连接失败，请检查网络设置"提示。MSW Mock模式下不依赖网络，不会出现此错误。

（2）**认证错误（401）：** JWT Token过期或无效时，后端返回401。前端拦截器自动清除token和用户信息，跳转到登录页面，显示"登录已过期，请重新登录"。

（3）**权限错误（403）：** 用户访问无权限的资源时返回403。前端显示具体提示："无权限执行此操作，请确认当前账号拥有相应权限"。

（4）**业务错误（500）：** 包括活动不存在、报名已满、签到距离超限、密码错误等。后端通过Result.error返回具体错误消息，前端直接展示。

（5）**地图加载错误：** 天地图Key失效或网络不通时，MapLibre的error事件捕获瓦片加载失败，前端在控制台输出警告但不影响其他功能。

（6）**AI服务错误：** 智谱API不可用时，后端自动降级到本地随机模板引擎。前端仍可正常使用AI生成功能，只是生成内容由模板引擎提供。

（7）**数据库连接错误：** MySQL不可用时，Spring Boot启动失败并输出详细错误日志。前端Mock模式下不需要数据库。

### 6.2 补救措施

（1）**数据备份：** 建议定期使用mysqldump导出数据库备份，存储在安全位置。

（2）**降级方案：** AI功能设计了"API优先+智能提取+本地模板"三层降级架构。地图底图加载失败时不影响表单等其他功能。前端Mock模式保证在任何情况下都能进行功能演示。

（3）**错误恢复：** 数据库操作使用Spring @Transactional事务管理，出现异常时自动回滚。报名操作使用事务保证signed_count与报名记录的一致性。

（4）**前端容错：** 活动创建页面的模板加载和AI调用使用try/catch包裹，失败时不影响页面其他功能。所有API请求通过Axios拦截器统一处理异常，不出现未捕获的错误。

### 6.3 系统维护设计

（1）**配置管理：** 系统关键配置（数据库连接、JWT密钥、AI Key、GeoServer地址等）集中在application.yml中，通过Spring @Value注入，修改配置后重启服务即可生效。前端环境变量（天地图Key、Mock开关等）集中在.env.development中，通过Vite的import.meta.env读取。

（2）**日志管理：** 后端使用SLF4J+Logback日志框架，MyBatis-Plus配置输出SQL日志（stdout），便于调试。关键操作（登录、创建活动、审核报名）通过OperationLog表记录。

（3）**数据库迁移：** 所有DDL变更通过init.sql脚本管理，使用INFORMATION_SCHEMA动态检测字段是否存在，确保多次执行不报错。新加入的开发者执行init.sql即可获得完整的数据库结构。

（4）**代码规范：** 前端使用TypeScript提供类型安全，Vue 3采用Composition API+script setup语法。后端遵循Spring Boot三层架构（Controller-Service-Mapper），统一使用Lombok减少样板代码。Git提交遵循feat/fix/docs约定式提交格式。

（5）**版本管理：** 代码托管于GitHub（wwaaaizai/volunteer-gis），采用分支开发策略。master分支为稳定版本，各开发者从master创建feature分支，完成开发后通过Pull Request合并。
