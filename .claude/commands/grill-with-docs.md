## 执行流程

1. **先探索再提问** — 如果问题可以通过探索代码库回答，先探索代码库，而不是问用户
2. **逐一追问** — 每次只问一个问题，等待反馈后再继续
3. **对每个问题提供你的推荐答案** — 不要只列出选项，给出你认为的最佳方案

## 领域意识

### 文档结构
本项目的上下文文档存放在 `doc/` 目录下：
```
/
├── CLAUDE.md              ← 项目指令 + AI 开发规范
├── doc/
│   ├── AI_CONTEXT.md      ← 项目全貌速览（充当 CONTEXT.md 的角色）
│   ├── API_REFERENCE.md   ← API 参考文档
│   ├── DATABASE.md        ← 数据库参考文档
│   └── FRONTEND_STRUCTURE.md ← 前端结构文档
├── docs/
│   └── adr/               ← 架构决策记录（按需创建）
├── volunteer-server/      ← 后端
└── volunteer-web/         ← 前端
```

如果尚未创建 `docs/adr/` 目录，在需要第一个 ADR 时再创建。

## 质询会话中的行为

### 挑战术语一致性
当用户使用的术语与 `doc/AI_CONTEXT.md` 中的定义冲突时，立即指出。
例："你的语境中'报名'指的是 Signup 记录，但你似乎指的是 Activity 参与者——是哪个？"

### 厘清模糊语言
当用户使用模糊或过载的术语时，提出精确的规范术语。
例："你说的'活动'——是 Activity 实体还是 Activity 参与记录？这是不同的概念。"

### 用具体场景讨论
当讨论领域关系时，用具体场景进行压力测试。设计能探测边界情况的场景。

### 交叉验证代码
当用户陈述某事的运作方式时，检查代码是否同意。如果发现矛盾，立刻指出。

### 即时更新文档
当一个术语被确定时，立即更新 `doc/AI_CONTEXT.md`。不要批量处理——在发生时就记录。

### 谨慎创建 ADR
只有同时满足以下三个条件时才创建 ADR：
1. **难以逆转** — 以后改变主意的成本很大
2. **没有上下文会很令人惊讶** — 未来的读者会疑惑"为什么这样做？"
3. **真正取舍的结果** — 有真正的替代方案，并且你因具体原因选择了其中一个

ADR 格式见 `doc/ADR-FORMAT.md`。
