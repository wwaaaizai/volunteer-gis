## 2. AI 开发规范 — 文档同步维护

### 2.1 强制原则

AI 在修改代码时**必须同步维护文档**，不允许"只改代码、不更新文档"。

### 2.2 文档变更触发规则

| 代码变更类型 | 必须更新的文档 | 更新内容 |
|-------------|---------------|----------|
| 新增/修改/删除 API 端点 | `doc/API_REFERENCE.md` | 端点、HTTP 方法、路径、请求参数、响应格式、错误码 |
| 修改数据库表结构（增/删/改列、索引） | `doc/DATABASE.md` **+ `db/init.sql`** | 表 DDL、索引定义、状态枚举值 **+ 增量 ALTER TABLE 迁移块** |
| 新增/修改/删除前端路由或页面组件 | `doc/FRONTEND_STRUCTURE.md` | 路由表、组件树、交互流程 |
| 修改技术栈或引入新依赖 | `doc/AI_CONTEXT.md` | 技术栈表格、子系统结构、文件路径映射 |
| 修改核心业务规则（签到距离、时长计算等） | `doc/AI_CONTEXT.md` + 对应模块文档 | 业务规则章节 |
| 重大架构决策（引入 Redis、切换 PostGIS 等） | `docs/adr/` 新建 ADR | 按模板记录决策 |
| 新增/删除文档文件本身 | `CONTEXT-MAP.md` | 文档索引表 |

### 2.3 文档更新原则

1. **增量更新**: 只修改变更涉及的文档部分，不重写整个文档
2. **保持一致**: 文档内容必须与代码实现完全一致，不允许文档滞后
3. **交叉引用**: 修改一个文档时，检查其他文档是否有相关引用需要同步更新
4. **术语统一**: 使用 `docs/markdown文档生成格式规范.md` 中定义的术语一致性表
5. **及时性**: 文档更新与代码修改在同一个 commit/操作中完成

### 2.3.1 数据库迁移规范（多设备协作关键）

**这是多设备协作下最容易出问题的环节，必须严格遵守。**

每次修改数据库表结构（增列、删列、改列、新增表）时，必须同时完成以下 3 件事：

1. **修改 `db/init.sql` 中的 `CREATE TABLE` DDL**（如果是新增表）
2. **在 `db/init.sql` 末尾追加增量 `ALTER TABLE` 迁移块**（如果是修改已有表）

   ```sql
   -- Phase X 增量迁移：描述
   ALTER TABLE `table_name`
     ADD COLUMN IF NOT EXISTS column_name TYPE COMMENT '说明';
   ```

   迁移块必须使用 `IF NOT EXISTS`，确保在新设备上可重复执行。如果 MySQL 版本不支持 `IF NOT EXISTS`，至少写上完整的 ALTER TABLE 语句并注释说明。

3. **更新 `doc/DATABASE.md`** 中对应表的列定义、状态枚举

**新增表时必须额外更新**：
- `doc/DATABASE.md` 添加新表的完整列定义和索引
- `CONTEXT-MAP.md` 文档索引中更新表的数量引用
- CLAUDE.md 第 3 节中 `DATABASE.md` 的"N 张表"引用

**未遵守此规范的后果**：其他协作者拉取代码后，后端启动会因缺少列/表而报错（SQLSyntaxErrorException），需要手动修复数据库 schema。



### 2.4 文档创建规则

- 新文档放入 `doc/`（技术参考）或 `docs/`（项目管理）
- 新文档必须同步更新 `CONTEXT-MAP.md` 索引
- 遵循 `docs/markdown文档生成格式规范.md` 的格式要求
